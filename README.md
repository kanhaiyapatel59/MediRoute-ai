# Redmi / Mediroute AI — Complete Detail

> **Purpose of this document**: Provide a single, detailed, and copy-paste friendly guide for the **Mediroute AI** setup and usage with a focus on **Redmi device behavior**, configuration expectations, and operational troubleshooting.

---

## 1) What this project does

**Mediroute AI** is a full-stack application (frontend + backend) that:

- Lets users authenticate (patient / hospital / admin roles depending on your setup).
- Stores and scores hospitals.
- Handles emergency requests.
- Generates **AI-driven ambulance/hospital recommendations**.
- Provides route mapping / fallback flows for cases where routing fails.

---

## 2) Project structure (high level)

- **backend/**: Node.js + Express + MongoDB (Mongoose)
  - `server.js` — server bootstrap
  - `routes/*Routes.js` — API endpoints
  - `controllers/*Controller.js` — business logic
  - `models/*Model.js` — MongoDB schemas
  - `services/*Service.js` — scoring / recommendation logic
  - `middleware/*Middleware.js` — auth and request handling
  - `seedAdmin.js` — seeds admin/hospital defaults

- **frontend/**: React + Vite
  - `src/pages/*` — UI pages
  - `src/api/axiosInstance.js` — Axios config & base URL / interceptors
  - `src/components/ProtectedRoute.jsx` — route guarding

## 5) Redmi / mobile behavior guidance

### 5.1 HTTPS and mixed content

- If you serve the frontend via **http** but call **https** APIs (or vice versa), Redmi/Android Chrome may block requests.
- Ensure both are consistently reachable (prefer HTTPS for production).

### 5.2 Location / mapping permissions (if used)

If you rely on location or map routing:

- On Redmi, ensure **Location Services** are enabled.
- Browser permissions:
  - Allow location permission for the site.
  - If the app uses embedded map, ensure the map provider permissions aren’t blocked.

### 5.3 Network instability and retry

- Cellular networks can drop during heavy API calls.
- Use idempotent requests where possible.
- For emergency flows, ensure the backend returns meaningful errors quickly.

### 5.4 Performance constraints

- Redmi devices may have lower CPU/RAM than desktops.
- Avoid large synchronous computations in the UI.
- Keep API payloads small.

## 6) Route fallback (MapRouteFallback) — expected purpose

The frontend page:

- **`frontend/src/pages/MapRouteFallback.jsx`**

is typically responsible for showing a fallback UI when routing fails or cannot be computed.

### Common fallback scenarios

- Route provider returns an error
- Missing coordinates
- User denies location permission
- Network error/timeouts

### Redmi-specific checks

- Confirm the fallback page triggers under the same conditions as on desktop.
- Confirm mobile layout does not hide key CTA buttons (e.g., “Retry”, “Call Ambulance”).

## 7) AI recommendation — how it decides hospitals

AI recommendations are generated by backend services, typically:

- `backend/services/aiRecommendationService.js`
- `backend/services/hospitalScoringService.js`

### Typical inputs

- Emergency type / severity
- Hospital availability
- Hospital scoring signals (distance, rating, capacity, hub flags, etc.)

### Critical fallback requirement

If there is **no active emergency hub hospital** (`isActiveEmergencyHub: true`), recommendations may fail.

**This project includes a TODO** to ensure:

- When no hub hospitals exist, the system should **fallback** (e.g., choose best non-hub hospitals or relax filters) rather than returning an empty response.

## 10) Troubleshooting checklist (Redmi focused)

### A) API calls failing only on Redmi

- Check mixed content (http/https)
- Check CORS policy on backend
- Check Axios base URL
- Confirm device time/date is correct (JWT validity)

### B) Empty recommendation list

- Confirm at least one hospital exists in DB
- Confirm emergency hub hospital flag is set (or fallback logic is implemented)

### C) App crashes on register/login

- Confirm frontend is sending `req.body` properly
- Backend should return 400 on missing payloads (project TODO)

### D) Location/map issues

- Verify location permissions
- Test with Wi‑Fi vs cellular
- Try disabling battery saver for the browser
